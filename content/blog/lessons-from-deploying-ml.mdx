## The Naive Start

Like most ML engineers, I started with a Jupyter notebook. The model worked great — 96% accuracy on my test set. I was ready for production.

Or so I thought.

## Lesson 1: The Model is the Easy Part

The model took me 2 weeks. Making it actually useful in production took 3 months. Here's what I needed:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY model/ ./model/
COPY app/ ./app/

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

A Dockerfile was just the beginning. I needed:
- **API layer** (FastAPI with proper error handling)
- **Input validation** (Pydantic schemas)
- **Caching** (Redis for frequent predictions)
- **Monitoring** (Prometheus metrics + Grafana dashboards)
- **Logging** (structured JSON logs for debugging)

## Lesson 2: Input/Output Contracts Matter

In notebooks, you pass a DataFrame and get predictions. In production, you need rigorous contracts:

```python
from pydantic import BaseModel, Field
from typing import List, Optional

class PredictionRequest(BaseModel):
    features: List[float] = Field(..., min_length=10, max_length=100)
    threshold: Optional[float] = Field(default=0.5, ge=0.0, le=1.0)

class PredictionResponse(BaseModel):
    prediction: int
    confidence: float
    model_version: str
    latency_ms: float

@app.post("/predict", response_model=PredictionResponse)
async def predict(request: PredictionRequest):
    start = time.perf_counter()
    result = model.predict(request.features)
    latency = (time.perf_counter() - start) * 1000

    return PredictionResponse(
        prediction=int(result > request.threshold),
        confidence=float(result),
        model_version="v1.2.0",
        latency_ms=round(latency, 2),
    )
```

## Lesson 3: Model Versioning is Non-Negotiable

I learned this the hard way when a colleague deployed an old model and we spent 3 hours debugging why predictions were wrong.

```
models/
├── v1.0.0/
│   ├── model.pkl
│   └── metadata.json
├── v1.1.0/
│   ├── model.pkl
│   └── metadata.json
└── v1.2.0/
    ├── model.pkl
    └── metadata.json
→ latest -> v1.2.0
```

Every model gets:
- A version number (semver)
- Training metadata (dataset hash, hyperparameters, date)
- Validation metrics
- A symlink `latest` pointing to the current production model

## Lesson 4: Data Drift Will Happen

The model that performed at 96% in testing dropped to 82% after 3 months in production. The data distribution had shifted.

The fix: monitor input distributions and alert when they drift.

```python
from scipy.stats import ks_2samp
import numpy as np

def detect_drift(reference: np.ndarray, current: np.ndarray, threshold: float = 0.05):
    statistic, p_value = ks_2samp(reference, current)
    return {
        "drift_detected": p_value < threshold,
        "p_value": float(p_value),
        "statistic": float(statistic),
    }
```

Set up a weekly job that compares recent inputs against the training distribution. When drift is detected, retrain the model.

## Lesson 5: CI/CD for ML is Different

You can't just run `npm test` and deploy. ML CI/CD needs:

1. **Data validation** — check that new data matches the schema
2. **Model evaluation** — run the test suite against the candidate model
3. **A/B testing** — route 5% of traffic to the new model
4. **Rollback** — instant revert if metrics degrade

```yaml
# .github/workflows/deploy-model.yml
name: Deploy Model
on:
  push:
    branches: [main]
    paths: ["models/**"]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Test model
        run: python tests/test_model.py
      - name: Evaluate metrics
        run: python scripts/evaluate.py --model models/latest
      - name: Deploy
        run: python scripts/deploy.py --model models/latest
```

## The Final Stack

After 3 months of iteration, the production stack looked like this:

| Component | Technology |
|-----------|-----------|
| API | FastAPI + Uvicorn |
| Container | Docker + Docker Compose |
| Model Registry | Local filesystem + S3 backup |
| Monitoring | Prometheus + Grafana |
| Logging | ELK Stack (Elasticsearch + Logstash + Kibana) |
| CI/CD | GitHub Actions |
| Drift Detection | Custom (scipy + scheduled job) |
| Caching | Redis |

## Key Takeaway

Deploying ML to production is 20% data science and 80% software engineering. The skills that mattered most weren't model architecture — they were API design, monitoring, error handling, and DevOps.

If you're building your first production ML system, start with the infrastructure first. The model can always be improved later, but unreliable infrastructure will kill user trust immediately.
