## Why RAG?

Large Language Models are powerful, but they have a fundamental limitation: they only know what they were trained on. If you ask about proprietary documents, recent events, or niche domain knowledge, they hallucinate or say "I don't know."

Retrieval-Augmented Generation (RAG) solves this by giving the model access to your own data at inference time. Instead of fine-tuning (which is expensive and brittle), you retrieve relevant chunks from a vector database and inject them into the prompt.

This post walks through a production-grade RAG pipeline I built for a client who needed to query thousands of technical PDFs.

## Architecture Overview

```
User Query → Embedding Model → Vector Search → Retrieve Chunks → LLM → Response
                                                      ↕
                                                Vector DB
                                                (Pinecone)
```

## Step 1: Chunking Strategy

The quality of your RAG system depends almost entirely on how you chunk your documents. Too small and you lose context. Too large and you dilute relevance.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=128,
    separators=["\n\n", "\n", ".", " ", ""],
    length_function=len,
)

chunks = text_splitter.split_documents(documents)
print(f"Created {len(chunks)} chunks")
```

Key decisions:
- **Chunk size**: 512 tokens — balances precision with context
- **Overlap**: 128 tokens — prevents information loss at boundaries
- **Separators**: prioritize paragraph breaks, then sentences

## Step 2: Embedding & Indexing

I used OpenAI's `text-embedding-3-small` model (1536 dimensions) and Pinecone as the vector store.

```python
from langchain_openai import OpenAIEmbeddings
from langchain_pinecone import PineconeVectorStore

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

vectorstore = PineconeVectorStore.from_documents(
    documents=chunks,
    embedding=embeddings,
    index_name="technical-docs",
)
```

### Why Pinecone?
- Serverless — no infrastructure to manage
- Built-in hybrid search (dense + sparse)
- Namespaces for multi-tenant isolation

## Step 3: Retrieval with Re-ranking

A single vector search isn't enough. I implemented a two-stage retrieval:

1. **Initial retrieval**: Fetch top-20 chunks using cosine similarity
2. **Re-rank**: Use a cross-encoder to score relevance

```python
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def retrieve(query: str, k: int = 5):
    initial = vectorstore.similarity_search(query, k=20)
    pairs = [(query, doc.page_content) for doc in initial]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(initial, scores), key=lambda x: x[1], reverse=True)
    return [doc for doc, _ in ranked[:k]]
```

This re-ranking step improved relevance by ~30% in our A/B tests.

## Step 4: Generation with Context

The final step is feeding the retrieved chunks into the LLM with a carefully crafted prompt:

```python
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_template(
    """You are a technical documentation assistant.
Answer the question based on the provided context.
If the context doesn't contain the answer, say so — don't make it up.

Context:
{context}

Question: {question}
Answer:"""
)

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
chain = prompt | llm
```

## Results

The final pipeline achieved:
- **Latency**: ~1.2s per query (retrieval + generation)
- **Accuracy**: 94% on our test set of 200 questions
- **Coverage**: Reduced hallucination rate from 22% to 3%

## Key Takeaways

1. **Chunking matters more than the model** — experiment with different strategies
2. **Re-ranking is worth the latency** — it dramatically improves result quality
3. **Prompt engineering is critical** — the right prompt structure makes or breaks the system
4. **Monitor everything** — log queries, retrieved chunks, and responses for continuous improvement
